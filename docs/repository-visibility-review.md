# GitHub リポジトリ公開範囲の検討メモ

- 検討日: 2026-07-15
- 対象リポジトリ: `toyokan/t-pod`
- 公開サイト: `https://events.toyokan.co.jp/`
- 現在の構成: GitHub の公開リポジトリを、GitHub Pages の `main` / `/ (root)` から公開し、会社のサブドメインを `CNAME` と DNS で割り当て
- 開発環境: Claude Code および Codex を利用

## 結論

現状の用途と構成では、**公開リポジトリを維持して差し支えない**。

ただし、この判断は「リポジトリに置くものは、Git の履歴を含めてすべて外部公開されてもよい情報に限る」という運用を前提とする。未発表イベント、社内限定資料、個人の非公開連絡先、認証情報などを同じリポジトリで扱う必要が生じた場合は、非公開リポジトリへの移行または公開物との分離を再検討する。

## 判断理由

### 公開リポジトリが適している点

- 本サイトは、イベント情報を不特定多数へ提供する静的 PWA である。
- HTML、JavaScript、Service Worker、イベント JSON はブラウザへ配信されるため、サイト利用者から取得・閲覧できる情報である。
- サーバー側処理、データベース、ログイン機能、決済機能を持たない。
- GitHub Pages を簡潔に運用でき、GitHub Free の組織でも利用できる。
- カスタムドメインの利用と、リポジトリの公開・非公開は別の論点である。会社のサブドメインを使っていること自体は、公開リポジトリを不適切にする理由にはならない。

### 公開に伴う注意点

- 現在のブランチ直下公開では、サイト用ファイルだけでなく、README、運用メモ、テンプレート、Claude Code／Codex 向け指示書なども公開リポジトリから閲覧できる。
- コミット履歴には変更内容、作者名、コミット用メールアドレスが残る。
- 削除したファイルや過去の記述も Git 履歴に残り得る。秘密情報を後から削除するだけでは十分ではない。
- 公開されたコードやデータは、第三者が複製・参照できる。リポジトリを後から非公開にしても、既に作られたコピーや公開 fork は回収できない。
- イベント JSON に記載した事務局の電話番号やメールアドレスは、サイト表示だけでなく生の JSON としても取得できる。

## 2026-07-15 時点の確認結果

現行ファイルおよび到達可能な Git 履歴を、代表的な秘密情報のパターンで確認した。

- API キー、パスワード、秘密鍵、アクセストークンに該当する情報は見つからなかった。
- `.env`、秘密鍵、証明書などに相当するファイル名は履歴上で見つからなかった。
- イベントの問い合わせ先として、事務局・担当者の電話番号とメールアドレスが含まれている。公開サイトに掲載する情報として承認済みであることを、イベントごとに確認する必要がある。
- コミット作者の個人メールアドレスが Git 履歴に含まれている。今後のコミットでは GitHub の `noreply` アドレスまたは会社が承認したアドレスを推奨する。
- Excel 入力テンプレートと運用資料がリポジトリに含まれている。現時点で機密情報とは判断しないが、サイト利用者への配布が不要なら、将来的に Pages の公開成果物とリポジトリ内の運用資料を分離する余地がある。
- `AGENTS.md`、`CLAUDE.md` は Codex／Claude Code が設計方針や禁止事項を共有するための開発用指示書であり、本プロジェクトでは開発に必要なファイルとして意図的に管理している。これらが存在すること自体は、公開リポジトリを不適切にする理由にはならない。

この確認は代表的な文字列・ファイル名による点検であり、機密情報が存在しないことを完全に保証するものではない。新しいイベントを追加する際の確認を継続する。

## 公開を継続する場合の運用ルール

1. `events/*.json` には、公開承認済みの情報だけを記載する。
2. 未発表のイベント情報を公開リポジトリへ先行投入しない。
3. API キー、パスワード、秘密鍵、アクセストークン、`.env`、社内 URL、非公開資料をコミットしない。
4. 配布資料の実ファイルは置かず、公開設定済みの外部 URL を記載するという現行方針を維持する。
5. 電話番号、個人名付きメールアドレス、講師・参加者情報は、掲載許可と公開範囲を確認する。
6. コミット用メールアドレスは GitHub の `noreply` または会社承認済みアドレスを使う。
7. GitHub の Secret scanning、Dependabot、ブランチ保護または ruleset を、利用可能な範囲で有効にする。
8. 秘密情報を誤ってコミットした場合は、単に削除コミットを追加するのではなく、直ちに認証情報を失効・再発行し、必要に応じて履歴除去を行う。

## Claude Code／Codex 向け指示書の扱い

`AGENTS.md` や `CLAUDE.md` は開発手順を安定させるために有用なので、公開リポジトリでも保持してよい。ただし、これらも一般公開される文書として扱う。

- API キー、トークン、パスワード、環境変数の実値を記載しない。
- 社内システムの非公開 URL、サーバー構成、個人情報、契約情報を記載しない。
- 社内だけで使う認証手順やアクセス権の詳細を記載しない。
- 公開するとセキュリティ上不利になる検知・防御の詳細を記載しない。
- AI ツールへ渡してよい情報と、渡してはいけない情報を会社の規程に合わせる。
- 指示書の変更も通常のコード変更と同様にレビューする。外部からの Pull Request を受け付ける場合は、指示書に不審な命令が追加されていないか特に確認する。
- AI ツールが生成した内容は、そのまま公開せず、担当者が個人情報、著作権、正確性、外部リンクを確認する。

会社固有の非公開手順が必要になった場合は、公開版の指示書へ追記せず、リポジトリ外の社内文書または非公開リポジトリで管理する。

## 非公開化を再検討する条件

次のいずれかが発生した場合は、リポジトリの非公開化または構成分離を再検討する。

- 開催発表前のイベントを GitHub 上で準備する必要がある。
- 社内の制作資料、校正資料、契約情報、原価、参加者名簿などを扱う。
- 外部に公開しない連絡先や資料 URL を扱う。
- 会社の情報セキュリティ規程で、業務リポジトリの公開が認められない。
- ソースコードや制作履歴を会社の知的財産として非公開管理する方針になった。
- 外部の共同編集者について、公開リポジトリでは不十分なアクセス制御が必要になった。

## 非公開化する場合の推奨構成

非公開化が必要になった場合は、次のどちらかを採用する。

### A. 非公開リポジトリから公開 Pages を配信

GitHub Pro、Team、Enterprise 等、非公開リポジトリから GitHub Pages を利用できるプランを使用する。リポジトリが非公開でも、公開設定された Pages サイトとブラウザへ配信されるファイルは公開情報である点に注意する。

### B. 制作元と公開成果物を分離

- 非公開リポジトリ: 未公開イベント、運用資料、テンプレート、生成処理
- 公開対象: 確定済みの HTML、JSON、Web Manifest、画像など、サイト配信に必要なファイルのみ

GitHub Actions などで公開成果物だけを Pages 用ブランチまたは別リポジトリへ配置する。この方式は公開範囲を明確にできる一方、現在の「ビルド工程なし」という簡潔さは失われる。

## 非公開化作業時の重要な注意

- GitHub Free の組織では、公開リポジトリを非公開にすると Pages が停止する場合があるため、契約プランと Pages の利用条件を先に確認する。
- カスタムドメインを設定したまま Pages を停止・移行すると、設定状況によってはドメイン乗っ取りのリスクが生じる。リポジトリの可視性を変更する前に、Pages 設定と DNS の切替手順を決める。
- リポジトリを非公開にしても、既存の公開 fork や第三者が取得したコピーは非公開にならない。

- 新しい種類のデータや機能を追加するとき
- フォーム、認証、決済、アクセス解析などを導入するとき
